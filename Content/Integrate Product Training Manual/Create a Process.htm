1.2 Create a Process

A Process is a series of posting steps:

  • It defines how Integrate takes data from the DSP database, and transfers it to the business systems.
  • It serves to link the data in a column in a view with a field on a template assigned to the process.

Each posting step is a template that is added to the process.

Processes are grouped into categories for organizational purposes. A process must have at least one template. Multiple templates can be added to a process for more complex posts/loads. Each template is tied to one, or multiple views.

When a template is added to a process, Integrate imports the loops and settings from the template into the process, and fields are mapped to columns in views. Depending on the template type, Integrate may perform field mapping automatically. Fields can be mapped manually or use the Auto Generate Database Objects feature. The Auto Generate Database Objects is a feature that quickly configures process template loops, relationships, field mappings and creates views and tables in the designated database.

  1. Select txFinishedMaterials_JDEInt from View Name list box.

NOTE:Integrate filters the views based on the prefix tx and suffix Int. If the view does not display in the list box, the naming convention is incorrect. If a view is not selected, it can be auto generated based on the recording. Auto Generate Database Objects creates tables and views in the designated database for the process template loops. Integrate creates each view with the name tx[Process Name] {Template Name}Int and creates each table with the name tt[Process Name] {Template Name]_Upload.

  1. Click Documentation tab.
  1. Enter notes about the process in Notes field.
  1. Click Advanced Properties tab to view the Execution Options.
  1. Verify Asynchronous is selected from Post Execution Method list box.
  1. Verify 0 displays in Delay Between Templates field.
  1. Change Delay Between Records to 250.

NOTE:A process that uses the Asynchronous post execution method, the option that is selected by default, executes the post for each template sequentially without waiting for the template post to return before executing the next template post. Asynchronous posting may result in locking, especially when posting a combination of GUI and BDC templates in the same process. A process that uses the Synchronous post execution method executes the post for each template, but waits for the post to return before executing the next post. Synchronous is a slower execution method but prevents most record locking situations.

 

NOTE:The default Delay Between Templates value is 0 seconds, which indicates there is no delay between the execution of templates during a post. Enter a value greater than 0 if posting a multiple template process that contains transactions that take some non-trivial time to Save and exit. This situation is more common when working with GUI Script templates than with BDC Script templates.

 

NOTE:The Delay Between Records field displays the amount of time, in milliseconds, that passes after the processing of one record is complete and the processing of the next record can begin. This setting should be used to avoid data locking issues. The recommended time is from 250 (.25 seconds) to 1000 (1 second). This setting is used for processes based on a BDC Script, GUI Script, or BAPI/RFC template.
